# План: Крипто-платежи и рейк

**Создан:** 2026-07-14
**Статус:** ВСЕ решения согласованы с владельцем (2026-07-14), открытых вопросов нет; реализация не начата
**Ветка:** будет создана при старте реализации

## Контекст и цель

Перевод игры на реальные деньги: депозиты/выводы исключительно в крипте
и монетизация через рейк с кэш-столов. План фиксирует согласованные
решения (пег, линейка столов, структура рейка) и архитектуру платежей,
чтобы реализация не пересматривала базовые константы.

**Дисклеймер (принято осознанно):** покер на реальные деньги — азартная игра.
Риски: лицензирование, AML-политики процессоров, политика Telegram по
гемблингу в Mini Apps (возможна блокировка приложения по жалобе).

## Принятые решения

### A. Пег фишки — КОНСТАНТА, менять после запуска нельзя

- **1 фишка = 0.01 USDT** (фишка = цент). Весь движок остаётся целочисленным.
- USDT (TON, 6 decimals): `usdtRaw = chips * 10_000`. Ни одного float на пути денег.
- На входе принимаем **только USDT** (TON как валюту депозита не берём —
  курсовой риск; добавить можно позже с мгновенным свапом).
- Мин. депозит: **$5** (500 фишек). Мин. вывод: **$10** (комиссии + анти-абуз).

### B. Линейка столов и бай-ины

- Бай-ин везде **диапазоном 40–100BB** (вместо текущего фиксированного 50BB).
  Требует изменения `TableConfig` (`buyIn` → `minBuyIn`/`maxBuyIn`) и валидации.
- Добавляется **funnel-стол $0.01/$0.02** (блайнды 1/2 фишки, бай-ин 80–200 фишек
  = $0.80–$2.00) — онбординг для первого депозита, не источник выручки.
- Итоговая линейка:

| Стол | Блайнды (фишки) | Блайнды ($) | Бай-ин ($, 40–100BB) |
|---|---|---|---|
| Funnel (новый) | 1/2 | $0.01/$0.02 | $0.80–$2.00 |
| Beginner ×2 | 5/10 | $0.05/$0.10 | $2.00–$5.00 |
| Standard ×2 | 10/20 | $0.10/$0.20 | $4.00–$10.00 |
| Pro | 25/50 | $0.25/$0.50 | $10.00–$25.00 |
| High Stakes | 100/200 | $1/$2 | $40.00–$100.00 |

- Глубину стека НЕ режем: 25BB-столы вырождают игру в push/fold,
  повышают дисперсию и снижают рейк (банки ограничены стеками).

### C. Рейк

- **5% от банка** (в конфиге `rakeBps: 500` — целое, без float), кэп в BB
  снижается с ростом лимита:

| Стол | Рейк | Кэп | Кэп в $ |
|---|---|---|---|
| Funnel | 5% | 3 BB | $0.06 |
| Beginner | 5% | 4 BB | $0.40 |
| Standard | 5% | 4 BB | $0.80 |
| Pro | 5% | 3 BB | $1.50 |
| High Stakes | 5% | 2.5 BB | $5.00 |

- **No flop, no drop:** раздача закончилась префлопом → рейк 0.
- **Uncalled bet не рейкуется:** `rakeableAmount = totalPot − uncalled`,
  где `uncalled = max(totalBet) − второй по величине totalBet` среди внёсших.
  Считать именно через разницу вкладов, НЕ по структуре потов (пот с одним
  eligible-игроком может быть честно выигран у сфолдивших — он рейкуется).
- **Кэп применяется к раздаче целиком**, не к каждому сайд-поту:
  `rakeTotal = min(floor(rakeable * rakeBps / 10000), capChips)`, затем
  списание распределяется по потам пропорционально размеру (floor по каждому,
  остаток — с мейн-пота). Иначе мультивей-олл-ины дают 2–3 кэпа с раздачи.
- Округление рейка — всегда `floor` (банки < 20 фишек дают 0 автоматически,
  отдельный минимальный порог не нужен).
- **Funnel-стол — С рейком** (не безрейковый): рейк удорожает чип-дампинг
  между аккаунтами (депозит на A → слив B → вывод). Это анти-абуз мера.
- Конфиг в `TableConfig`: `rakeBps: number`, `rakeCapBB: number`.

### D. Платёжный провайдер

- **MVP: Crypto Pay API (@CryptoBot).** Invoice → webhook `invoice_paid` →
  зачисление; вывод — метод `transfer`. Интеграция минимальная, кастодиальный
  посредник — осознанный размен. Риск: может отказать гемблингу по своим правилам.
- **Позже: TON Connect** (USDT-jetton, свой hot-кошелёк + индексация toncenter/tonapi)
  как второй канал.
- Выводы на старте — **ручное подтверждение админом** (очередь заявок `pending`),
  лимит суммы/сутки обязателен и после автоматизации.

### E. Леджер (обязателен ДО подключения денег)

- Модель `Transaction` (append-only): `id`, `userId`, `type`
  (`deposit | withdrawal | buyin | cashout | rake | bonus | adjustment`),
  `amount` (signed), `balanceAfter`, `externalId` (invoice id CryptoBot,
  **@unique** — идемпотентность против дублей webhook'ов), `status`
  (`pending | completed | failed`), `meta` (handId, tableId), `createdAt`.
- Все изменения баланса — только `UPDATE ... SET balance = balance + $1
  WHERE balance + $1 >= 0` + вставка Transaction в одной Prisma-транзакции.
  Никаких read-modify-write.
- Рейк пишется в леджер (`type: 'rake'`, системный house-аккаунт или null-userId)
  и в `HandHistory.rake: Int`. В meta рейк-записи — раскладка по участникам
  банка пропорционально вкладу (задел под рейкбек без миграций).
- **Инвариант для крон-проверки:** `депозиты − выводы − Σ балансов −
  фишки в активных играх = накопленный рейк`. Расхождение = алерт.

### F. Прозрачность в UI

- Рейк показывать явно: `PotResult` дополнить полем `rake`
  («Pot $2.40, rake $0.12»); в инфо стола — структуру
  («5%, cap 4BB, no flop no drop»). Скрытый рейк для крипто-аудитории =
  обвинения в скаме.
- Балансы в UI дублировать в $ («1000 фишек = $10»).

## Точки врезки рейка в код

Две точки в `server/Game.ts`, рейк снимается ДО распределения:

1. **Win-by-fold** (~строка 628, `winner.chips += totalWon`) —
   только если `stage >= flop` (no flop no drop) и за вычетом uncalled.
2. **Шоудаун** (цикл по `this.pots`, ~строки 798–826) — перед
   `Math.floor(pot.amount / winners.length)`.

**Ловушка движка:** uncalled bet НЕ возвращается игроку отдельно — он проходит
через `calculatePots` (~строки 566–589) как пот с единственным eligible-игроком
и «выигрывается» им самим. Наивный рейк по каждому поту возьмёт 5% с денег,
которые игроку просто вернулись. Отсюда формула через разницу вкладов (см. C).

### G. Play-money и daily bonus — УДАЛЯЮТСЯ (решено 2026-07-14)

- Вариант А: всё реальное, один `balance`. Daily bonus задумывался для
  упрощения тестирования MVP — в реальной экономике не нужен и опасен
  (раздача выводимых денег каждые 24ч).
- Удалить: логику daily bonus в `UserRepository`, UI кнопки бонуса,
  `lastDailyRefill` можно оставить в схеме мёртвым полем до отдельной чистки.
- Дефолт `balance` для нового юзера: **0** (не 1000!) — фишки появляются
  только через депозит.
- Онбординг-лестница без бесплатной песочницы: funnel ($0.80) → Beginner ($2) → выше.

### H. House-аккаунт — системная строка User (решено 2026-07-14)

- Служебный User с `HOUSE_TELEGRAM_ID = 0` (боты занимают отрицательный
  диапазон; реальные Telegram id начинаются с 1 — коллизия невозможна).
  Рейк-транзакции пишутся на него, `balance` накапливает прибыль.
- Инвариант упрощается: `депозиты − выводы = Σ всех балансов + фишки в играх`
  (рейк — не особый случай, а баланс house).
- Исключить house из лидербордов/статистики/поиска игроков (~3-4 WHERE-фильтра),
  защитить от логина.
- **Вывод прибыли — только через админку** (никогда напрямую через UI CryptoBot,
  иначе крон-сверка разойдётся): раздел «House» → проверка атомарным
  `UPDATE ... WHERE balance - $1 >= 0` (технически нельзя вывести деньги
  игроков) → `withdrawal`-транзакция house → Crypto Pay `transfer` на личный
  telegram_id владельца → `completed`. AdminAuditLog автоматически.
- На house также относятся: комиссии Crypto Pay с депозитов, сетевые комиссии
  выводов игроков (`type: 'fee'`), ручные компенсации (`adjustment`).
  Баланс house = чистая прибыль (рейк − комиссии − компенсации).
- Решить при реализации: комиссию депозита берёт дом (игроку зачисляется
  полный номинал) или игрок — зафиксировать явно в коде зачисления.

### I. Анти-абуз на выводах вместо формального KYC (решено 2026-07-14)

- **Документный KYC не делаем**: без гемблинг-лицензии провайдеры верификации
  не онбордят, а самодельная проверка документов — бесполезный театр.
  Идентичность = Telegram-аккаунт; AML на уровне денег частично закрывает
  CryptoBot (кастодиальный, со своим мониторингом).
- Главный риск — транзит/отмывание (депозит → минимальная игра или слив
  фишек сообщнику → вывод «выигрыша»). Меры:
  1. **Порог активности:** вывод доступен после **50 сыгранных рук с момента
     последнего депозита** (фильтр от чистого транзита, не wagering).
  2. **Флаги в админ-очереди выводов** (подсветка, НЕ автобан): вывод > 90%
     недавнего депозита при < 100 рук; систематический слив крупных банков
     одному оппоненту (чип-дампинг, запрос по HandHistory); группа аккаунтов,
     играющих только друг с другом.
  3. **Лимиты:** $500/сутки на игрока при ручном подтверждении;
     после автоматизации выводов — до $100 авто, выше — ручная проверка.
- Все цифры (50 рук, $500, $100) — в конфиге, не в коде; крутим по факту.
- НЕ делаем: гео/IP-блокировки (VPN-аудитория), автобаны по эвристикам
  (на малых объёмах ложные срабатывания дороже ручного взгляда).

## Открытые вопросы

Нет — все решения приняты 2026-07-14, план готов к реализации с фазы 1 (леджер).

## Порядок реализации (фазы)

1. **Леджер** — модель Transaction, переписать все мутации баланса
   (buyin/cashout/bonus) через транзакции. Без этого деньги не подключаем.
2. **Рейк** — конфиг в tables.ts, врезка в Game.ts (обе точки), запись в
   леджер/HandHistory, отображение в UI. Тестируется на play-money.
3. **Бай-ин 40–100BB + funnel-стол** — TableConfig, валидация, UI выбора суммы.
4. **Депозиты (Crypto Pay)** — invoice, webhook с идемпотентностью, зачисление.
5. **Выводы** — заявки, админ-очередь подтверждения, transfer, лимиты.
6. **Крон-проверка инварианта** + алерты.
