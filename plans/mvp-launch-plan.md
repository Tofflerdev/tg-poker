## Блокеры запуска (Critical — без этого нельзя идти в прод)

### 1. Деплой и HTTPS

Telegram Mini App требует HTTPS. Без этого приложение не откроется в Telegram.

**Что нужно:**
- VPS или PaaS (Railway, Render, VPS + nginx)
- SSL-сертификат (Let's Encrypt)
- Обратный прокси nginx → Node.js сервер
- Настроить CORS под продовый домен
- Настроить `BOT_TOKEN` и `DATABASE_URL` в prod-окружении

**Файлы для создания:**
- `Dockerfile` — для упаковки сервера
- `nginx.conf` — для раздачи клиента и проксирования API/WebSocket
- `.env.production` — пример prod-конфига

---

### 2. Telegram Bot + Mini App URL

**Что нужно:**
- Зарегистрировать бота через @BotFather
- Установить Mini App URL: `/setmenubutton` или `/newapp`
- Проверить, что initData валидируется корректно в prod-режиме

**Текущая проблема:** В dev-режиме `auth.ts` принимает пустой initData. В проде нужно убедиться, что `NODE_ENV=production` и dev-режим выключен.

---

### 3. Reconnect-логика (восстановление состояния)

**Проблема:** Если у игрока оборвалась связь в середине руки — он теряет сессию. Сервер делает автофолд и возвращает фишки, но игрок не может «вернуться» за стол.

**Что нужно реализовать:**
- При переподключении (новый socket.id, тот же telegramId) — восстановить игрока за его столом
- Слать ему актуальный `gameState` после reconnect
- Не трогать его фишки до таймаута (например, 60 секунд)

**Затронутые файлы:** `server/index.ts`, `server/models/Table.ts`

---

### 4. Стабильность при перезапуске сервера

**Проблема:** Фишки хранятся in-memory в объекте `Game`. При краше сервера — все игровые сессии теряются, фишки «пропадают».

**Что нужно:**
- Фиксировать buy-in сразу при входе (уже сделано ✅)
- При крашe/рестарте — при следующем подключении игрок видит свой баланс в БД (уже частично сделано ✅)
- Добавить: сохранение состояния «игрок за столом» в БД, чтобы после рестарта не было ситуации «деньги списаны, за стол не вернулся»

**Минимальное решение:** Добавить в БД поле `currentTableId` и `currentChips`. Сбрасывать при выходе/краше. При рестарте — возвращать зависшие фишки на баланс при следующем логине.

## Важные задачи (Should Have — для нормального запуска)

### 5. Обработка ошибок на клиенте

**Текущая проблема:** При разрыве соединения, ошибках сервера — нет внятных сообщений и retry-логики.

**Что нужно:**
- Экран «Reconnecting...» с индикатором при потере соединения
- Error boundary для React-компонентов
- Toast-уведомления для server-side ошибок (недостаточно баланса, стол занят и т.д.)
- Автоматический retry для socket подключения

**Затронутые файлы:** `client/src/App.tsx`

---

### 6. Валидация ставок и edge cases

**Текущие риски:**
- Игрок может отправить `raise` с суммой ниже минимального рейза — нужна серверная проверка
- Игрок с 0 фишками не должен оставаться за столом
- Одновременные действия (двойной клик по кнопке) — дедупликация на сервере

**Что нужно:**
- В `Game.ts` — ужесточить валидацию всех действий
- На клиенте — дизейблить кнопки после нажатия до получения ответа

---

### 7. Лидерборд

**Зачем:** Ключевой инструмент удержания игроков. Без него игра быстро надоедает.

**Что нужно реализовать:**
- Таблица топ-10 по балансу / выигрышам
- Отдельная страница `LeaderboardPage.tsx`
- Серверный endpoint: выборка топ пользователей из БД
- Обновление раз в минуту или по запросу

**Новые файлы:**
- `client/src/pages/LeaderboardPage.tsx`
- Добавить в `UserRepository.ts`: метод `getLeaderboard(limit: number)`
- Добавить Socket-событие: `getLeaderboard` / `leaderboardData`

---

### 8. Защита от злоупотреблений

**Проблемы:**
- Частый join/leave для «промывки» фишек
- Спам в чате
- Попытки переиграть ставки через дублирование событий

**Минимальные меры:**
- Rate limit на Socket.io события (пакет `socket.io-rate-limiter` или ручная реализация)
- Debounce join/leave: нельзя покинуть стол и зайти снова в течение 5 секунд
- Максимальная длина сообщения в чате: 200 символов
- Запрет на вход за стол, если баланс < buy-in (уже частично есть ✅)

---

### 9. История рук (базовая)

**Зачем:** Нужна хотя бы для отладки и поддержки пользователей.

**Минимальный вариант:**
- Добавить в схему Prisma таблицу `hand_results`
- Сохранять: tableId, handNumber, winners, pot, timestamp
- Не хранить карты — только итоги
- API для получения последних 20 рук за конкретным столом

**Схема:**
```prisma
model HandResult {
  id        Int      @id @default(autoincrement())
  tableId   String
  pot       Int
  winners   Json     // [{telegramId, amount}]
  createdAt DateTime @default(now())
}
```
