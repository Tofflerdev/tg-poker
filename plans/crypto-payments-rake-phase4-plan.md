# План: Фаза 4 — Депозиты (Crypto Pay) + house-аккаунт

**Создан:** 2026-07-21
**Родительский план:** `plans/crypto-payments-rake-plan.md` (фаза 4, §D/§E/§H)
**Статус:** согласован с владельцем 2026-07-21, реализация не начата
**Ветка:** `feat/deposits-phase4` (создать при старте)

## Цель

Подключить реальные депозиты в USDT через Crypto Pay API (@CryptoBot) и
ввести house-аккаунт, на который оседает рейк. После этой фазы фишки могут
появляться на балансе не только из play-money, но и из оплаченного инвойса.

## Согласованные решения (владелец, 2026-07-21)

1. **Комиссию Crypto Pay платит игрок** — зачисляем НЕТТО (пришедшая сумма
   за вычетом комиссии провайдера). House на депозитах комиссию не несёт;
   отдельная `fee`-строка на house за депозит НЕ пишется.
2. **Play-money НЕ трогаем в этой фазе.** Daily bonus и дефолт `balance` (1000)
   остаются — нужны для тестирования. §G (удаление бонуса, `balance=0` по
   умолчанию, обнуление прод-балансов) переносится в отдельный шаг
   **«MVP-release cleanup»** и делается перед реальным запуском, НЕ здесь.
3. **house-аккаунт (§H) делаем в этой фазе** — он ортогонален play-money и
   нужен, чтобы рейк перестал быть null-scoped и у выводов прибыли (фаза 5)
   появился источник.
4. **bot bankroll account (§K) делаем в этой фазе** — отдельный системный
   аккаунт-флоат для бай-инов ботов, которых на начальном этапе подсаживают
   к живым столам для массовки. Отделён от house (house = прибыль с рейка,
   bankroll = флоат под массовку, его дрейф = стоимость/выручка массовки).
   Пополняется **внешне владельцем** (свои деньги через админку). При нехватке
   флоата на бай-ин бот **не сажается + алерт** (никакого овердрафта).

## Открытые вопросы

Нет. Комиссия и судьба play-money решены выше.

## Блок 0 — подготовка (пег + конфиг)

**0.1 Env-переменные** (`.env.example`, чтение в `server/index.ts`):
- `CRYPTO_PAY_TOKEN` — токен приложения @CryptoBot (обязателен для платежей).
- `CRYPTO_PAY_TESTNET` — `true` → база `https://testnet-pay.crypt.bot/api`,
  иначе `https://pay.crypt.bot/api`.
- Депозиты активны только если токен задан; иначе socket-событие депозита
  возвращает ошибку «payments disabled» (dev/play-money режим работает без токена).

**0.2 Пег-утилита** `server/payments/peg.ts`:
- `MIN_DEPOSIT_CHIPS = 500` ($5), константа пега `USDT_RAW_PER_CHIP = 10_000`.
- `chipsToUsdt(chips): string` → строка для API (`amount: '5'`), без float в деньгах.
- `usdtToChips(amountStr): number` → `floor` пришедшей USDT-суммы в фишки
  (1 фишка = $0.01). Округление вниз — недоплату не дарим.
- Юнит-тесты: пег туда-обратно, floor на некратных суммах, отсечка < min.

## Блок §H — house-аккаунт (делать ПЕРВЫМ)

**H.1 Сид house-юзера** (`server/db/UserRepository.ts` + вызов при старте в `index.ts`):
- `ensureHouseUser()` — идемпотентный upsert `telegramId = 0`,
  `displayName = 'House'`, `balance` не сбрасывать при update (только create → 0).
- Константа `HOUSE_TELEGRAM_ID = 0` в одном месте (напр. `server/payments/house.ts`
  или рядом с конфигом), реиспользуется фильтрами.

**H.2 Переписать `recordRake`** (`UserRepository.ts:95`):
- Сейчас пишет `userId: null, balanceAfter: null`. Теперь — одной Prisma-транзакцией:
  `UPDATE users SET balance = balance + rake WHERE telegram_id = 0`, затем строка
  `Transaction{ type:'rake', userId: house.id, balanceAfter, meta }`.
- Инвариант плана §H: `депозиты − выводы = Σ балансов + фишки в игре`
  (рейк перестаёт быть особым случаем — это баланс house).

**H.3 Исключить house из витрин:**
- Лидерборды / статистика / поиск игроков — добавить `telegramId <> 0`
  (грепнуть по местам, где раньше уже фильтруется `isBot`).
- `middleware/auth.ts` — запретить логин/авторизацию под `telegramId = 0`
  (реальные Telegram id ≥ 1, боты отрицательные — коллизии нет).

**H.4 Тесты:** рейк кредитует house-баланс; house не появляется в лидерборде;
инвариант сходится на синтетической раздаче с рейком.

## Блок §K — bot bankroll account (флоат под массовку)

**Контекст.** Сейчас `addBots` (`admin/adminMutations.ts:229`) сажает ботов через
`table.addPlayer(..., table.config.maxBuyIn, ...)` — фишки берутся из воздуха,
баланс не списывается, боты `balance 0`, `isBot=true` и **намеренно обходят
checkpoint/refund** (`d188c47`). За реальными столами это ломает инвариант:
человек, выигравший банк у бота, получает реально выводимые фишки, не покрытые
деньгами. Значит фишки, которые бот приносит на стол, должны быть реальным
профинансированным флоатом. Отдельный аккаунт держит этот флоат.

**K.1 Сид bankroll-юзера** (`UserRepository.ensureBankrollUser()` + старт в `index.ts`):
- Зарезервированный `BOT_BANKROLL_TELEGRAM_ID` (константа рядом с `HOUSE_TELEGRAM_ID`).
  Боты занимают отрицательный диапазон seat-идентичностей — bankroll id тоже
  зарезервировать и **исключить из `acquireBotIdentity`** (`server/bot/…`), чтобы
  ни один бот-сит не занял его. Отдельный от house (`telegramId=0`).
- Идемпотентный upsert; `balance` при update не трогаем (только create → 0).

**K.2 Пополнение (внешнее, владельцем)** — админ-действие `topUpBankroll(amount)`:
- Атомарно: `UPDATE balance += amount WHERE telegram_id = BANKROLL`, строка
  `Transaction{ type:'adjustment', userId: bankroll.id, amount:+, balanceAfter, meta:{reason:'bankroll_topup'} }`.
  Это **внешний приток денег** → на money-in стороне инварианта (см. ниже).
- AdminAuditLog автоматически (через `runWithAudit`, как остальные мутации).

**K.3 Бай-ин бота из bankroll** — новый путь при посадке бота (`addBots`):
- Заменить «фишки из воздуха» на атомарное списание с bankroll:
  `tryDecrementBalance(BANKROLL, buyIn, meta:{tableId,seat,bot:true})` (гардовый
  `WHERE balance >= buyIn`). Если списание не прошло (нехватка флоата) —
  **бота НЕ сажаем**, инкремент счётчика пропусков + алерт в лог/админку.
- `buyIn` = `clampBuyIn(...)` стола (как у людей). Ledger-строка `buyin` на bankroll.
- ⚠️ Не переиспользовать человеческий session-checkpoint (`currentChips` и т.п.):
  bankroll — не сит, а бот по-прежнему не чекпоинтится. Фишки бота живут в Game-стейте.

**K.4 Кэшаут/баст бота обратно в bankroll** — хук при снятии бота со стола:
- В `removeBots` / выходе бота на границе раздачи прочитать **текущий стек бота
  из Game-стейта** и вернуть его на bankroll: `UPDATE balance += stack`, строка
  `cashout` на bankroll (`amount:+stack`, `meta:{tableId, bot:true}`). Баст = стек 0,
  строку с нулём не пишем (как `refundCurrentChips` при 0).
- Убедиться, что путь срабатывает во ВСЕХ ветках ухода бота (ручной remove,
  авто-удаление на границе раздачи, дисконнект таблицы) — иначе флоат «утекает»
  в вечно-занятые фишки и инвариант разъезжается.

**K.5 Инвариант с bankroll.** Обобщённая форма (для крон-проверки, фаза 6):
```
Σ(депозиты + внешние пополнения bankroll) − Σ(выводы)
  = Σ всех балансов (игроки + house + bankroll) + фишки во всех играх (вкл. ботовые)
```
- Ботовые фишки на столах — часть «фишек в играх». Т.к. боты не чекпоинтятся,
  крон должен брать их стек из **живого Game-стейта**, а не из `currentChips` в БД.
- Дрейф bankroll (баланс − суммарные пополнения) = чистая стоимость/выручка массовки.

**K.6 Витрины/защита** (как house): исключить bankroll id из лидербордов/статистики/
поиска (`telegramId <> BANKROLL`) и запретить логин под ним (`auth.ts`).

**K.7 Тесты:** бай-ин бота списывает bankroll; нехватка флоата → бот не сел + флаг;
кэшаут/баст возвращает стек; инвариант с bankroll сходится на синтетической
раздаче человек-vs-бот (человек выиграл → bankroll упал ровно на переданные фишки).

## Блок 4 — депозиты (Crypto Pay)

**4.1 `CryptoPayClient`** `server/payments/cryptoPay.ts` (чистый `fetch`, без SDK):
- База из env (testnet/prod), заголовок `Crypto-Pay-API-Token: <token>`.
- `getMe()` — проверка токена при старте сервера (лог ok/fail, платежи не валят старт).
- `createInvoice({ amount, payload })` → POST `/createInvoice`
  (`asset:'USDT'`, `amount`, `payload:<txId>`, `allow_comments:false`).
  Возвращает `invoice_id` и URL оплаты (`mini_app_invoice_url` / `bot_invoice_url`).
- `verifyWebhookSignature(rawBody, header)` — HMAC-SHA256 тела ключом
  `sha256(CRYPTO_PAY_TOKEN)`, сравнение `timingSafeEqual` с заголовком
  `crypto-pay-api-signature`.

**4.2 Создание инвойса** — socket-событие `createDeposit(amountChips)` (`index.ts`):
- Валидация: целое, ≥ `MIN_DEPOSIT_CHIPS`, платежи включены.
- Записать `Transaction{ type:'deposit', status:'pending', userId, amount:+chips,
  externalId: String(invoice_id) }` (externalId @unique → защита от дублей).
- Вернуть клиенту URL оплаты.
- **Порядок:** сначала `createInvoice`, потом запись pending со свежим `invoice_id`
  (иначе нет externalId). Если запись упала — инвойс осиротеет, это ок (не оплачен).

**4.3 Webhook** — `app.post('/api/crypto/webhook', ...)` (`index.ts`):
- **ВАЖНО:** нужен доступ к сырому телу для подписи. Смонтировать
  `express.raw({ type: 'application/json' })` ТОЛЬКО на этом пути (глобальный
  `express.json()` уже стоит на `:91` — не ломать его, повесить raw до/локально).
- Проверить подпись → 401 при провале (не раскрывать причину).
- Обрабатывать только `update_type === 'invoice_paid'`.
- **Идемпотентное зачисление НЕТТО** одной Prisma-транзакцией:
  1. Найти `deposit`-строку по `externalId = invoice_id`.
     - Нет строки → 200 (чужой/устаревший инвойс), лог.
     - `status === 'completed'` → 200 и выход (повторная доставка webhook).
  2. Посчитать `netChips = usdtToChips(paid − providerFee)` из payload
     (комиссию платит игрок → зачисляем нетто). Если поле комиссии в payload
     отсутствует — уточнить формат ответа Crypto Pay и зафиксировать в коде.
  3. `UPDATE balance += netChips` игроку, перевести строку в `completed`,
     проставить `balanceAfter`, дописать факт (`meta`: paid, fee, rate).
  4. Ответить 200 быстро (Crypto Pay ретраит при не-2xx).
- Уведомить клиента по socket (`depositCredited`) — если юзер онлайн, обновить баланс.

**4.4 UI депозита** (`client/`, неон-стиль):
- Экран/модалка «Пополнить»: ввод суммы в $, readout в фишках (пег), кнопка
  ≥ $5. Открытие URL через `WebApp.openInvoice` (нативный поток CryptoBot)
  или `openLink`.
- Статус «ожидание оплаты» → обновление баланса по событию `depositCredited`.
- Дублировать баланс в $ (план §F: «1000 фишек = $10»).

## Тестирование

**Юнит (server):**
- peg round-trip + floor + min-отсечка.
- webhook: валидная подпись зачисляет один раз; повторная доставка не
  дублирует (idempotency по externalId); битая подпись → 401.
- recordRake кредитует house; house скрыт из витрин.

**Прод-плейтест (testnet Crypto Pay):**
- Реальный тестовый инвойс → оплата → проверка: баланс вырос на нетто,
  ровно одна `completed`-строка, инвариант §H сходится (крон-скрипт как в фазах 1-3).

## Порядок работ

1. Блок 0 (пег, env) — фундамент, тесты.
2. Блок §H (house) — рейк переезжает на house, фильтры, тесты.
3. Блок §K (bot bankroll) — сид, пополнение, бай-ин/кэшаут ботов через флоат, тесты.
4. Блок 4.1–4.3 (клиент API, инвойс, webhook) — ядро депозитов.
5. Блок 4.4 (UI).
6. Тесты + testnet-плейтест.

§H и §K — независимые системные аккаунты, можно делать параллельно; §K опирается
на §H только идейно (общий паттерн сид+фильтры), кодовой зависимости нет.
