### AVATAR — Custom Avatar System

- [x] **AVATAR-01** — 20 animal avatar WebP assets shipped as hashed Vite bundle assets. **Phase 2 Plan 02-02 + separate commit.**
- [x] **AVATAR-02** — New users assigned random avatar on first auth, atomically (UPDATE WHERE avatarId IS NULL). **Phase 2 Plan 02-02.**
- [x] **AVATAR-03** — User can re-pick avatar from Profile Avatar tab; choice propagates everywhere. **Phase 2 Plan 02-06.**
- [x] **AVATAR-04** — SeatsDisplay renders each seat's avatar via manifest (initial-letter fallback). **Phase 2 Plan 02-02.**

### PROFILE — Profile Expansion & Hand History

- [x] **PROFILE-01** — Profile page shows balance, handsPlayed, handsWon, totalWinnings, biggestPot + display name, avatar, daily-bonus eligibility. **Phase 2 Plan 02-06.**
- [x] **PROFILE-02** — Hand history persisted on hand completion via async batched write queue (HandHistoryQueue). **Phase 3 Plan 03-02.**
- [x] **PROFILE-03** — Profile shows last 50 hands (date, table, hole cards at showdown only, board, result, net delta). **Phase 3 Plan 03-04.**
- [x] **PROFILE-04** — Hand history older than 90 days removed by retention job; no opponent hole cards at non-showdown. **Phase 3 Plan 03-04.**

### DEPOSIT — Deposit Stub

- [x] **DEPOSIT-01** — Main menu shows first-position "Deposit" block in Neon Strip. **Phase 2 Plan 02-04.**
- [x] **DEPOSIT-02** — Tapping Deposit opens in-app "Coming soon" page; no external links, no payment SDK. **Phase 2 Plan 02-04.**

### GAME — Game Table Enhancements

- [x] **GAME-01** — Redundant table/phase label (top-left) and pot label (top-right) removed from game room. **Phase 2 Plan 02-07.**
- [x] **GAME-02** — On every player action, floating "action bubble" appears over seat (Fold / Check / Call N / Bet N / Raise to N / All-in) with FIFO queueing. **Phase 3 Plan 03-03.**
- [x] **GAME-03** — Bubble duration ~800–1000 ms minimum; respects `prefers-reduced-motion`. **Phase 3 Plan 03-03.**
- [x] **GAME-04** — `Game.ts` exposes `setOnPlayerAction` and `setOnHandComplete` callbacks consumed by `server/index.ts`. **Phase 1 Plan 01-03.**

### RESILIENCE — Reconnect & Crash Safety

- [x] **RESILIENCE-01** — User persistence extended with `currentTableId`, `currentSeat`, `currentChips`, `sessionToken`, `disconnectedAt`, `lastSeenAt` via Prisma migration `v1_mvp_launch`. **Phase 1 Plan 01-02.**
- [x] **RESILIENCE-02** — Economic state written at hand boundaries via `onHandComplete`; mid-hand ephemeral state never persisted. **Phase 3 Plan 03-05.**
- [x] **RESILIENCE-03** — `TableManager`, `userStorage`, and socket mappings keyed by `telegramId`. **Phase 1 Plan 01-04.**
- [x] **RESILIENCE-04** — On reconnect, server re-verifies initData HMAC, emits `replacedBySession` to prior socket, sends full GameState snapshot. **Phase 4 Plan 04-06.**
- [x] **RESILIENCE-05** — Client shows "Reconnecting…" overlay; 30s mid-hand / 120s between-hands grace window. **Phase 4 Plan 04-05.**
- [x] **RESILIENCE-06** — On server boot, recovery module reseats or refunds persisted sessions. **Phase 4 Plan 04-04.**
- [x] **RESILIENCE-07** — Buy-in / cashout use atomic SQL (`UPDATE ... WHERE balance >= :n`); no double-spend. **Phase 4 Plan 04-01.**

### SECURITY — Production Auth Hardening

- [x] **SECURITY-01** — Dev-mode bypass gated on BOTH `ALLOW_DEV_AUTH=true` AND `NODE_ENV !== 'production'`. **Phase 1 Plan 01-05.**
- [x] **SECURITY-02** — On start: if `NODE_ENV=production` AND (`ALLOW_DEV_AUTH=true` OR `BOT_TOKEN` empty), process exits code 1. **Phase 1 Plan 01-05.**
- [x] **SECURITY-03** — HMAC comparison uses `crypto.timingSafeEqual`; never returns fabricated dev user on HMAC failure. **Phase 1 Plan 01-05.**
- [x] **SECURITY-04** — Telegram initData, sessionToken, raw telegramId scrubbed from Sentry events + logs + analytics via `beforeSend` hook. **Phase 5 Plan 05-02.**

### ADMIN — Hidden Admin Panel

- [x] **ADMIN-01** — Admins identified by `ADMIN_TELEGRAM_IDS` env allowlist; no isAdmin DB flag. **Phase 5 Plan 05-04.**
- [x] **ADMIN-02** — Admin on separate `io.of('/admin')` namespace with namespace-level HMAC + JWT middleware. **Phase 5 Plan 05-04.**
- [x] **ADMIN-03** — Admin UI is lazy-loaded client subtree with "ADMIN MODE" banner; not linked from player UI. **Phase 5 Plan 05-05.**
- [x] **ADMIN-04** — Admin dashboards: active tables, active users, economy, recent errors. **Phase 5 Plan 05-05.**
- [x] **ADMIN-05** — Admin can enable/disable table, drain, edit blinds/buy-in, kick user, ban user, grant balance delta. **Phase 5 Plan 05-04.**
- [x] **ADMIN-06** — Every admin mutation writes `AdminAuditLog` row BEFORE commit; failed audit write aborts mutation. **Phase 5 Plan 05-04.**

### TEST — UI Test Suite

- [x] **TEST-01** — Vitest + RTL + jsdom configured, runs from `npm test`; shared setup mocks Telegram.WebApp + Socket.io. **Phase 6 Plan 06-00.**
- [x] **TEST-02** — Every interactive UI element has at least one co-located `*.test.tsx` covering happy-path interaction. **Phase 6 Plans 06-01..06-03.**
- [x] **TEST-03** — Scenario tests: join-table, fold/call/raise, disconnect+reconnect UI, avatar selection, ToS gate, deposit navigation. **Phase 6 Plan 06-04.**
- [x] **TEST-04** — CI runs suite against prod-like Vite build; hard gate on push/PR to main. **Phase 6 Plan 06-05.**

### OBS — Observability

- [x] **OBS-01** — `@sentry/react` + `@sentry/node` initialized with shared DSN, environment/release tags, PII scrubber. **Phase 5 Plan 05-02.**
- [x] **OBS-02** — Sentry Replay enabled (errors sampled, privacy-masked). **Phase 5 Plan 05-02.**
- [x] **OBS-03** — PostHog Cloud on server + client; user identity = `sha256(telegramId)`; raw telegramId never leaves server. **Phase 5 Plan 05-02.**
- [x] **OBS-04** — Fixed event taxonomy: `user_signed_up`, `daily_bonus_claimed`, `table_joined`, `table_left`, `hand_completed`, `reconnect_succeeded`, `reconnect_failed`, `admin_action`, `error_shown`. **Phase 5 Plan 05-02.**

### COMPLIANCE — Responsible Gaming, ToS & Privacy

- [x] **COMPLIANCE-01** — ToS, Privacy Policy, and Responsible Gaming pages are static, reachable from main menu and settings, Neon Strip styled. **Phase 2 Plan 02-08.**
- [x] **COMPLIANCE-02** — New users must tap "Accept" on first-launch consent screen before `joinTable` honored; sets `tosAcceptedAt` + `tosVersion`. **Phase 2 Plan 02-08.**
- [x] **COMPLIANCE-03** — Existing users (grandfathered) see non-blocking ConsentBanner; not prevented from playing. **Phase 2 Plan 02-08.**
- [x] **COMPLIANCE-04** — Server-side `joinTable` handler rejects users with `tosAcceptedAt IS NULL` created after ToS gate shipped. **Phase 5 Plan 05-01.**
- [x] **COMPLIANCE-05** — RG page: virtual-chip disclaimer, "not for real money", daily-bonus-only economy, informational "take a break" guidance; no forced lockouts. **Phase 2 Plan 02-08.**
