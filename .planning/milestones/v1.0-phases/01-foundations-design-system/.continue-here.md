### AP-01 — Worktree branched from stale base (BLOCKING)

**What happened (2026-04-15, plan 01-02 first executor run):**

The gsd-executor agent was spawned with `isolation="worktree"`. On Windows, the worktree was created from a commit older than `EXPECTED_BASE` (before phase 01-01 had landed — neon.css and the tracked `.planning/` files did not yet exist in the worktree's working tree).

The safety check in the workflow is:
```
ACTUAL_BASE=$(git merge-base HEAD $EXPECTED_BASE)
if [[ $ACTUAL_BASE != $EXPECTED_BASE ]]; then
  git reset --soft $EXPECTED_BASE
fi
```

`git reset --soft` moves HEAD only. The working tree still lacked every file added since the old base (neon.css, STATE.md, ROADMAP.md, PROJECT.md, REQUIREMENTS.md, all research/*.md, 01-01-SUMMARY.md, 01-CONTEXT.md, 01-RESEARCH.md, and the 01-01 edits to App.tsx / GameControls.tsx / SeatsDisplay.tsx). The agent then `git add`-ed its intended changes and committed — but the commit also recorded the absent files as **deletions** and reverted 01-01's client edits.

Net effect if the worktree had been ff-merged back: loss of all phase-01-01 work and destruction of every planning artifact. Detected during merge-back because untracked `01-02-SUMMARY.md` collided — only then did the file-level diff expose the 15+ unintended deletions.

**Recovery:** files were salvaged via `git checkout worktree-agent-<id> -- <explicit-paths>` to pick only the intended `prisma/`, `.env.example`, and `01-02-SUMMARY.md` paths; corrupt branch and worktree were deleted.

**Structural mechanism to prevent recurrence (enforce before spawning any worktree executor):**

1. **Pre-create the worktree from the orchestrator**, not from the agent runtime:
   ```
   git worktree add -b <branch> <path> <EXPECTED_BASE>
   ```
   Pass the path to the agent instead of relying on `isolation="worktree"` auto-creation.

2. **If auto-creation is used**, the agent MUST verify the working-tree state, not just HEAD:
   ```
   git diff --quiet HEAD -- . && git diff --cached --quiet -- . \
     || { echo "worktree dirty relative to HEAD — abort"; exit 1; }
   ```
   Soft-reset alone is insufficient. A hard reset or fresh checkout from `EXPECTED_BASE` is required when the working tree does not match HEAD.

3. **Orchestrator post-agent check, before merge-back**: run
   ```
   git diff --stat <EXPECTED_BASE> <worktree-branch>
   ```
   and refuse to merge if the stat includes any **deletion** of a file that exists on main and is not listed in the plan's `files_modified`. Surface the mismatch to the user and salvage only the intended paths.

4. **Never `git merge --ff-only` a worktree branch blindly** — always inspect `git diff --name-status` first and confirm no unexpected deletions.

**Where this enforcement lives:** orchestrator step `execute_waves` in the execute-phase workflow, specifically the "Worktree cleanup" substep. This file documents the gap until the workflow is patched.
