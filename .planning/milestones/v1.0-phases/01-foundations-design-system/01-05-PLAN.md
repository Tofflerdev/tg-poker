<objective>
Harden the auth posture to fail-closed in production and remove every dev-user
fabrication path. Gate the dev bypass on BOTH `ALLOW_DEV_AUTH=true` AND
`NODE_ENV !== 'production'` (D-18). Add a fatal-exit boot check (D-19) that runs
BEFORE any listener binds. Replace string-equality HMAC comparison with
`crypto.timingSafeEqual` over equal-length Buffers (D-20). Delete every
`IS_DEV && createDevUser` fallback in `createUserFromInitData`.

`ALLOW_DEV_AUTH=true` (or ships with an empty `BOT_TOKEN`) must not serve traffic.
An HMAC failure must not silently become a dev login.

`server/index.ts`, `.env.example` documenting the new env var.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client initData → server auth | Untrusted input crosses the HMAC boundary. Must be validated constant-time. |
| Deploy env → running process | Env vars are the contract between the operator and the process. A mistake here (forgotten `ALLOW_DEV_AUTH=true`) must hard-fail before traffic is accepted. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-05-01 | E (Elevation of Privilege) | Deploying to prod with `ALLOW_DEV_AUTH=true` left over from staging | mitigate | `assertSafeBootOrExit` runs before any listener binds; `process.exit(1)` with a single FATAL stderr line. Task 3 boot-matrix proves all four corners. |
| T-01-05-02 | S (Spoofing) | HMAC failure silently downgraded to a fabricated dev user (current behavior) | mitigate | Every `createDevUser` fallback branch removed from `createUserFromInitData`. `validateInitData` returns null on any validation failure. |
| T-01-05-03 | I (Information Disclosure) | Timing side-channel on hash comparison reveals prefix match | mitigate | `crypto.timingSafeEqual` replaces `===` on Buffer-decoded values. Length check gates the call to prevent the `RangeError` pitfall. |
| T-01-05-04 | T (Tampering) | Malformed/oversize initData crashes the auth path (RangeError, JSON throw) | mitigate | `validateInitData` wraps the full body in try/catch; any throw → `return null`. Downstream handler emits `authError` and the socket is not privileged. |
| T-01-05-05 | I (Information Disclosure) | Auth logs echo raw initData (tokens, user name, etc.) | mitigate | Log lines reference failure class only; the initData string is never included. Phase 5 (SECURITY-04) adds PII scrubbing for Sentry/logs globally. |
| T-01-05-06 | D (DoS) | Attacker floods invalid HMACs to exhaust CPU | accept | `timingSafeEqual` is O(n) over 32-byte HMACs; socket.io rate-limits connections (default Express + socket.io behavior). Further hardening deferred; acceptable for MVP scale. |
| T-01-05-07 | R (Repudiation) | Operator claims the dev bypass was unreachable in prod | mitigate | FATAL stderr line is the audit trail. Task 3 boot-matrix script is checked in so any regression fails CI in Phase 6. |
</threat_model>
