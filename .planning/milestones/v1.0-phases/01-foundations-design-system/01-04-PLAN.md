Big-bang refactor of identity: `telegramId` (stringified) becomes the durable key across
`TableManager`, `userStorage`, `Player`, and every socket handler. `socket.id` becomes
a per-connection transport handle only. `socket.data.telegramId` is populated once at
auth and read everywhere downstream. Scaffold socket eviction for duplicate-session
connects (D-07) — Phase 4 fills in the `replacedBySession` payload + snapshot resume.

audit log, analytics identity) keys off telegramId. The refactor must land now so
Phase 3/4/5 don't relitigate it.

new eviction hook on TableManager wired into `server/index.ts` connection handler.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client socket → server handler | Each incoming socket event must resolve to the authenticated telegramId, not trust anything in the event payload. |
| Prior session → new session | A second connection with the same telegramId must not coexist with the first (split-brain / chip duplication risk). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-04-01 | S (Spoofing) | Client sends a telegramId in an event payload to impersonate another player | mitigate | Handlers MUST read identity from `socket.data.telegramId` (set only by the auth handler after HMAC validation). Task 3 bulk-audit enforces this; any payload-derived identity is rejected. |
| T-01-04-02 | T (Tampering) | Two concurrent sockets with the same telegramId create split-brain state (chip duplication at buy-in, conflicting actions) | mitigate | D-07 eviction hook fires on `setSocketForTelegram`: prior socket is disconnected before the new one is bound. Phase 4 expands to full `replacedBySession` + snapshot resume. |
| T-01-04-03 | E (Elevation of Privilege) | Unauthenticated socket reaches a game handler before auth completes | mitigate | Handlers short-circuit with `authError` if `socket.data.telegramId` is unset. Matches existing behavior; strengthened by removal of the `socket.id`-keyed fallback path. |
| T-01-04-04 | I (Information Disclosure) | Disconnect race leaves stale `socketByTelegram` entry → new connection may race the old disconnect | mitigate | Disconnect handler guards the clear with `if (getSocketIdForTelegram === socket.id)`; the new connection's `setSocketForTelegram` unconditionally wins via eviction. |
| T-01-04-05 | R (Repudiation) | Action attributed to wrong telegramId if socket.data.telegramId is stale after reconnect | mitigate | Eviction closes the prior socket before the new one can emit. `updatePlayerSocketId` keeps Player.socketId in sync. Phase 4 full-snapshot resume closes the remaining gap. |
</threat_model>
