# Phase 4: Resilience - Context

**Gathered:** 2026-04-22
**Status:** Ready for planning

<domain>
## Phase Boundary

Land the reconnect-resume + boot-time crash-recovery + atomic-balance loop on top of the substrate Phase 1 and Phase 3 already established.

**In scope:**
1. **Reconnect handshake (RESILIENCE-04):** Auth handler detects an existing seated session for the authenticated `telegramId`, emits the (renamed) `replacedBySession` event to any prior socket, evicts it, and pushes a personalized `tableJoined` + `state` snapshot (with own hole cards) to the new socket.
2. **Grace window state machine (RESILIENCE-05):** On `disconnect`, mark `disconnectedAt = now()` and arm a stage-aware grace timer. 30 s mid-hand / 120 s between hands. Mid-hand expiry → `sittingOut`; between-hands expiry → vacate seat + refund chips. Existing turn timer continues to run independently.
3. **Boot recovery (RESILIENCE-06):** On startup, every persisted session row (`currentTableId IS NOT NULL`) is refunded — `currentChips → balance`, all session columns cleared. Per-row try/catch loop calling the atomic D-D2 helper (no outer `$transaction` — the helper's conditional `updateMany` already provides per-row atomicity and idempotence). No reseat-as-sit-out path.
4. **Atomic balance SQL (RESILIENCE-07):** Buy-in deduction uses `updateMany WHERE balance >= n`; cashout / refund paths use `updateMany WHERE currentChips IS NOT NULL` for idempotency. `count === 0` ⇒ refuse with `tableError`.
5. **Client "Reconnecting…" overlay:** Delayed ~1.5 s after socket disconnect, full-screen Neon Strip with countdown, dismissed on next `tableJoined`. On grace expiry → non-blocking "sat out / removed" message with back-to-Table-List button.

**Out of scope (carried forward):**
- Mid-hand ephemeral state restore (hole cards stored in `Game` instance memory, regenerated by `Game.startNextHand()`) — Phase 3 D-17 keeps this principle.
- Action bubble replay on reconnect — Phase 3 deferred (forward-only).
- `sessionToken` column writes — schema column stays dormant in v1; reconnect re-runs HMAC every time.
- Sentry / PostHog instrumentation of `reconnect_succeeded` / `reconnect_failed` events — Phase 5 (OBS-04 owns the taxonomy).
- Server-side ToS gate on `joinTable` — Phase 5 (COMPLIANCE-04).
- Admin "kick user" path that uses similar eviction primitives — Phase 5 (ADMIN-05).

</domain>

<decisions>
## Implementation Decisions

### A. Reconnect Handshake & Snapshot (RESILIENCE-04)

- **D-A1:** **No `sessionToken` use in v1.** The schema column from Phase 1 D-14 stays in place but is never read or written by Phase 4. Every reconnect re-runs `validateInitData` HMAC (already happens — `server/index.ts:209`). Eviction is keyed solely by `telegramId` via the existing `tableManager.socketByTelegram` map (Phase 1 D-06). Matches success-criterion-1 wording: "session token only disambiguates" — with single-client-per-telegramId enforced by eviction, there is nothing to disambiguate.

- **D-A2:** **Reuse existing `state` + `tableJoined` events** on reconnect. No new `reconnectSnapshot` event type. The auth handler at `server/index.ts:208` is extended: after `setSocketForTelegram`, if `tableManager.getPlayerTable(telegramId)` returns a table, emit `tableJoined { tableId, seat, state: getStateForPlayer(telegramId) }` to the new socket and run `updateTableState(tableId)` to refresh other seats. `getStateForPlayer` already reveals own hole cards (Phase 1 / `server/Game.ts:getStateForPlayer`). Smallest blast radius, leverages everything already built.

- **D-A3:** **`replacedBySession` is a bare event with no payload** (rename of the Phase 1 D-07 placeholder `sessionReplaced`). After emit, server calls `socket.disconnect(true)`. The evicted client renders a static "You were logged in elsewhere" notice. No timestamp, no new-session-id leak.

- **D-A4:** **`GameState` payload is sufficient** for resume — no `lastAction` summary, no `serverNowMs` clock-drift correction in v1. `turnExpiresAt` already carries timer remaining. Bubble queue stays forward-only (Phase 3 deferred). Future enhancements (last-action context, drift correction) deferred to v1.1+ if mobile reconnect UX shows real drift.

### B. Grace Window Semantics (RESILIENCE-05)

- **D-B1:** **On `disconnect`, mark `disconnectedAt = now()` and leave the seat as-is.** No immediate sit-out, no immediate auto-fold. The existing `Game.TURN_TIME_LIMIT = 30000` continues to run independently — if it was the disconnected player's turn, the engine auto-folds them on the existing path (`server/Game.ts:startTurnTimer`). Arm a grace timer per D-B2.

- **D-B2:** **Grace duration set by snapshot of `game.stage` at disconnect.** If `stage === 'waiting'` or `stage === 'showdown'` (or there is no active hand) → 120 s timer. Else → 30 s timer. If the 30 s mid-hand timer is still running when the hand ends (other players finish without them), cancel it and re-arm a fresh 120 s between-hands timer. Tracks the actual game state honestly.

- **D-B3:** **Two-stage grace expiry:**
  - **Mid-hand (30 s) expiry** → set `sittingOut = true` on the player, clear `disconnectedAt`, KEEP `currentChips` / `currentTableId` / `currentSeat`. Player can reconnect later and `sitIn` to rejoin. Their seat remains held.
  - **Between-hands (120 s) expiry** → `tableManager.leaveTable(telegramId)`, refund `currentChips` to `balance` via the atomic refund path (D-D2), clear all session columns. Player must `joinTable` again to come back.
  - Matches success-criterion-3 / -4 wording "sat-out / vacated" — sit-out for short grace, vacate for long grace.

- **D-B4:** **Client "Reconnecting…" overlay UX:**
  - **Trigger:** socket `disconnect` event → start a 1500 ms debounce timer. On reconnect-before-debounce, never show overlay (avoids flicker on the WebSocket hiccups common on mobile Telegram).
  - **Render:** full-screen Neon Strip overlay (dark translucent `rgba(10,10,14,0.9)` + `backdrop-blur(12px)`), centered cyan-glowing "Reconnecting…" text with a depleting countdown ("25s remaining…").
  - **Countdown source:** client computes from grace duration locally (server doesn't push remaining time — the client knows the rules). Mid-hand vs between-hands inferred from last `GameState` (use `gameState.stage` from the snapshot before disconnect).
  - **Dismissed on:** next `tableJoined` event (means handshake completed and server pushed snapshot).
  - **On grace expiry** (no reconnect): swap overlay content to non-blocking "You were sat out — your seat is held" (mid-hand) or "You were removed from the table — chips returned to balance" (between-hands) with a single "Back to Tables" button that routes to Table List.

### C. Boot Recovery Policy (RESILIENCE-06)

- **D-C1:** **Always refund on boot.** Every persisted session (`currentTableId IS NOT NULL`) is treated as a clean reset — no reseat-as-sit-out branch. For each row: refund `currentChips` to `balance` and clear `currentTableId`, `currentSeat`, `currentChips`, `disconnectedAt`, `lastSeenAt`. Players reconnect after a server restart and see their full balance, then re-pick a table fresh. Honors the "never restore in-flight hand state" principle (Phase 3 D-17) most strictly and is robust to `config/tables.ts` edits between runs.

- **D-C2:** **Recovery runs after `setupTableEvents`, before `[Boot] HandHistoryQueue + retention job started`.** Position: inside the existing `setTimeout(..., 1000)` block at `server/index.ts:182`. Single `await recoverPersistedSessions()` call after the `tables.forEach((t) => setupTableEvents(t.id))` loop. No need to gate `server.listen()` — sessions are queued, not racing the listener.

- **D-C3:** **Stale `currentTableId` (no match in `PREDEFINED_TABLES`) → refund + warn.** Uses the same code path as the default. `console.warn('[BootRecovery] stale tableId %s for telegramId=%s — refunded', tableId, telegramId)`. Robust to renamed/removed tables.

- **D-C4:** **Per-row try/catch loop calling the D-D2 atomic helper — no outer `$transaction`.** `prisma.user.findMany({ where: { currentTableId: { not: null } } })` → for each row, call `UserRepository.refundCurrentChips(telegramId)` inside `try { ... } catch { /* log + continue */ }`. The helper's conditional `updateMany` (D-D2: `WHERE currentChips IS NOT NULL`) is itself atomic and idempotent in a single SQL round-trip, so wrapping it in an outer `prisma.$transaction([...])` would only add overhead without strengthening atomicity. The try/catch already bounds blast-radius (one bad row does not poison the sweep), and Phase 5 audit-logging can be added inside the loop body without needing a transaction. **Amendment 2026-04-29:** revised from "per-row Prisma `$transaction`" after plan-checker flagged the redundancy; the original intent (per-row atomicity + bounded blast-radius) is preserved by the try/catch + helper composition.

### D. Atomic Balance SQL (RESILIENCE-07)

- **D-D1:** **Prisma `updateMany` with conditional `where`** is the SQL form. No raw `$queryRaw` and no read-then-write transactions:
  ```ts
  const result = await prisma.user.updateMany({
    where: { telegramId, balance: { gte: amount } },
    data:  { balance: { decrement: amount } }
  });
  if (result.count === 0) { /* insufficient funds */ }
  ```
  Single round-trip, idiomatic Prisma, works with `@prisma/adapter-pg` (Prisma v7) already in use.

- **D-D2:** **Two guards by domain:**
  - **Buy-in path** (`server/index.ts:526` + the legacy `join` handler at `:739`): `WHERE balance >= n` — refuse on insufficient funds. Matches RESILIENCE-07 spec literal.
  - **Cashout / grace-expiry refund / boot-recovery refund:** `WHERE currentChips IS NOT NULL` idempotency guard so concurrent reconnect storms can't double-credit. Single SQL pattern:
    ```ts
    await prisma.user.updateMany({
      where: { telegramId, currentChips: { not: null } },
      data: {
        balance: { increment: <captured currentChips> },
        currentChips: null,
        currentTableId: null,
        currentSeat: null,
        disconnectedAt: null,
        lastSeenAt: null
      }
    });
    ```
    The captured `currentChips` value is read first (or returned via a separate `select`) before the update; the `IS NOT NULL` guard ensures the write only fires once.
  - **Daily bonus / hand-end winnings:** unchanged — already-safe paths (fixed-value set, server-authoritative computation respectively).

- **D-D3:** **Keep the existing pre-check at `server/index.ts:504`** (`user.balance < tableInfo.config.buyIn`) as a UX hint — it shortcuts the obvious case with no DB round-trip and emits `tableError` instantly. The atomic `updateMany` is the actual gate; its `count === 0` path ALSO emits `tableError` and aborts the join (rolling back the in-memory `joinTable` if it already happened). Defense in depth; closes Concern #5 from `.planning/codebase/CONCERNS.md`.

- **D-D4:** **Reuse existing `tableError` event** with the same `"Insufficient balance. Buy-in is N"` string the pre-check already emits. No new typed error event in v1. The two failure paths (pre-check + atomic SQL race) emit the same event so the client doesn't need to distinguish.

<specifics>
## Specific Ideas

- **`sessionToken` column stays dormant** — the schema column from Phase 1 stays in place for v1.1+ multi-device, but is unread/unwritten in v1.
- **30 s mid-hand grace duration is intentionally aligned with `Game.TURN_TIME_LIMIT = 30000`** — both timers expire on the same wall-clock budget when a player disconnects on their turn. This is not coincidence; it's the design's contract: "you have one turn-clock to come back, period."
- **120 s between-hands grace** is a generous mobile-Telegram allowance — long enough to switch apps, take a phone call, etc. — but short enough that other players don't perceive a stranded seat.
- **Boot recovery is always-refund** (no reseat) precisely because tables in this milestone are predefined and re-creatable; the in-memory `Game` instance is gone after restart anyway, so there is nothing to "reseat into" beyond an empty `Game` waiting for `addPlayer`. Refund-and-restart is the simplest correct semantics.
- **Atomic SQL form is `updateMany` not `$queryRaw`** to keep the Prisma boundary clean — no raw SQL maintenance burden in this codebase.
- **Same `tableError` string for both pre-check and atomic-SQL failure** — the user's experience is identical; the client doesn't need to learn a new error type.
- **Overlay debounce of 1500 ms** is calibrated for mobile Telegram WebSocket hiccups, which typically heal within ~300–800 ms — the debounce avoids overlay flicker on those.
